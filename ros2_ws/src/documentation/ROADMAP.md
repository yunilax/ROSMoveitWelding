# Дорожная карта: от прототипа к ТЗ

Документ описывает поэтапный план разработки программного обеспечения роботизированной сварочной системы в соответствии с [ТЗ](robot_welding_tz_final%20(5).docx).

**Репозиторий:** `ros2_ws`  
**Дата:** 2026-07-08  
**Горизонт:** ~14–16 месяцев (ПО)

---

## Точка старта (текущее состояние)

| Модуль ТЗ (§3.7) | Статус | Где в коде |
|---|---|---|
| Operator HMI | ~85% | `welding_web` |
| ICP Engine | ~70% | `welding_backend/icp.py`, `welding_web/PointCloudScanner.ts` |
| Kinematics Solver | ~60% | `welding_backend/kinematics.py`, `reachability.py` |
| Trajectory Planner | ~40% | `welding_backend/moveit_export.py`, `welding_web/MoveItExporter.ts` |
| Vision Engine | ~25% | симуляция сканирования |
| Robot Motion Engine | ~30% | `welding_bridge/weld_planner_node.py` |
| Calibration Manager | ~10% | константы в `WorkcellLayout.ts` / `kinematics.py` |
| Database Manager | 0% | — |
| Report Generator | 0% | — |
| Welding Controller Interface | 0% | — |
| Safety Supervisor | 0% | — |

**Рабочий цикл сегодня:** CAD → швы → sim scan → ICP → reachability → MoveIt JSON → rosbridge (stub).

## Фазы

### Фаза 0 — Стабилизация прототипа

**Срок:** 2–3 недели  
**Цель:** надёжный демо-цикл на Windows без ROS.

| # | Задача | ТЗ | Критерий готовности |
|---|---|---|---|
| 0.1 | Единый dev-стек | — | `start-dev.ps1` + документированный WSL-путь |
| 0.2 | Синхрон URDF ↔ UI ↔ backend | Ц.1 | Один источник констант workcell |
| 0.3 | IK с ориентацией TCP | Ц.7, Ц.9 | Учёт угла наклона горелки в export |
| 0.4 | NURBS-сглаживание траектории | Ц.8 | Плавные waypoints без скачков |
| 0.5 | Чистка временных patch-файлов | — | Репозиторий без `_patch*.py` |

**Результат:** оператор проходит полный цикл без ручных правок кода.

---

### Фаза 1 — Vision Engine

**Срок:** 1,5–2 месяца  
**Цель:** конвейер обработки облака точек (§3.5.1, Приложение Л, Ц.3–Ц.6).

| # | Задача | ТЗ | KPI |
|---|---|---|---|
| 1.1 | Конвейер фильтрации | Л, Ц.4 | SOR → Voxel → MLS |
| 1.2 | RANSAC базовых плоскостей | Ц.5 | Авто-определение базовых поверхностей |
| 1.3 | ICP v2 | Ц.3 | RMS ≤ 0,10 мм; запрет сварки при ошибке |
| 1.4 | Импорт PLY/PCD | §3.5 | Загрузка реального облака в UI |
| 1.5 | Seam refinement по scan | Ц.6 | Коррекция линии шва по фактической геометрии |
| 1.6 | Абстракция ScanProvider | §3.5 | Единый интерфейс sim / file / hardware |

**Архитектура:**

```
ScanProvider (sim | file | hardware)
    → FilterPipeline (SOR, Voxel, MLS)
    → SegmentEngine (RANSAC planes)
    → ICPEngine (Open3D, RMS gate)
    → SeamRefiner
    → TrajectoryPlanner
```

**KPI фазы:** ICP RMS ≤ 0,10 мм на тестовых деталях; распознавание швов ≥ 95% (путь к 98%).

### Фаза 2 — Машина состояний и автоматический цикл

**Срок:** 1 месяц  
**Цель:** логика комплекса как конечный автомат (§3.4, Приложение М).

| # | Задача | ТЗ |
|---|---|---|
| 2.1 | FSM в backend | OFFLINE → … → REPORT + EMERGENCY |
| 2.2 | Авто-переходы между шагами UI | §3.2.1 |
| 2.3 | Журнал событий | §3.2.1, FMEA (Прил. Т) |
| 2.4 | Полуавтоматический режим | §3.2.2 |
| 2.5 | Блокировка при ошибках | ICP fail, IK fail, collision |

**Состояния:**

```
OFFLINE → INITIALIZATION → READY
  → SCANNING → PROCESSING → TRAJECTORY
  → WELDING → INSPECTION → REPORT → READY
  ↓ (любая авария)
EMERGENCY
```

**KPI фазы:** ≥ 10 автоматических циклов подряд на sim-детали без вмешательства оператора.

## Приоритет ближайших работ

1. **Фаза 0** — закрепить текущий прототип.
2. **Фаза 1.1–1.3** — фильтры, RANSAC, ICP с RMS-gate.
3. **Фаза 2** — FSM и автоматический цикл.

## Риски

| Риск | Митигация |
|---|---|
| ROS не установлен в WSL | Документировать установку Jazzy; CI в WSL |
| ICP не достигает 0,10 мм на sim | Тестовый набор деталей с ground truth |
| Generic URDF ≠ реальный робот | Отдельный пакет `vendor_robot_description` в Фазе 4 |
| Сканер не выбран | Абстракция `ScanProvider` в Фазе 1 |
| Safety — аппаратная часть | Фаза 7: только ПО-интерфейс; SIL/PL — отдельный контур |
